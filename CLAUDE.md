# CLAUDE.md

## 项目定位

Yggdrasil — 自进化的 Agent 认知引擎。一棵活的、会呼吸的认知之树，将技能、知识、记忆统一为有机的认知原子网络，支持自主生长、强化、修剪、涅槃。

**设计哲学**：结构优于规模，生长必须伴随修剪，可回滚比永远正确更重要，Agent 与环境共同进化。

### 决策框架

做任何非平凡技术决策时按以下顺序展开：

1. **是否解决真实问题** — 不是假想需求，能说出具体触发场景、谁会受益
2. **至少 2 个方案 + trade-off** — 维度涵盖：复杂度、性能、可观测性、扩展性、迁移成本、对现有抽象的冲击
3. **识别关键约束** — 状态外置 / 优雅关闭 / 跨进程协作 / token 预算，是否有被破坏的
4. **明确退路** — 选择失败时如何回滚、如何降级、灰度策略

不要给出"唯一正确答案"，特别是在抽象层（Node / Edge / Retrieval / Season）的设计上。即使我催促也要先把 trade-off 摆出来。

## 架构分层

```
api (REST HTTP 接口，轻薄层) → jobs (后台任务：巡检、四季轮转) → core (核心能力：节点/边/检索/进化) → infra (基础设施：MySQL、日志、HTTP 客户端)
```

- `src/api/` — Quart web 层：中间件、路由、端点、请求校验
- `src/jobs/` — 后台任务：巡检调度、四季节律执行、健康检查
- `src/core/` — 核心能力：Yggdrasil 引擎、认知原子模型、检索算法、进化机制
- `src/infra/` — 基础设施：MySQL 连接池、日志、告警、HTTP 客户端、向量存储

### 分层原则

- **api 层禁止业务逻辑**：只做参数校验、调用 core，错误转换为 HTTP 响应
- **jobs 层是调度边界**：周期性任务在这里收敛，不承载核心业务逻辑
- **core 层是认知引擎中心**：Node/Edge/Retrieval/Evolution 在这里演进，对外提供能力而不暴露细节
- **infra 层只做基础设施**：不感知业务，不依赖上层

跨层调用方向是单向的（api → jobs → core → infra），反向依赖视为设计缺陷。

## 技术栈

- **Web 框架**: Quart（async Flask 等价，基于 Hypercorn ASGI）
- **数据库**: MySQL via aiomysql（连接池 + 原生 SQL，无 ORM）
- **DI**: dependency-injector (JSR-330 风格)
- **配置**: pydantic-settings，环境变量前缀 `YGGDRASIL_DB_`
- **可观测**: 自定义 LogService 异步缓冲写、prometheus-client 指标、TraceEventBus
- **Python**: >=3.11, asyncio

## 关键约定

### 认知原子模型

所有认知内容统一存储为节点，通过 `role` 区分功能类型：

| role | 对应类型 | 说明 |
|------|---------|------|
| capacity | Skill | 可执行能力，包含触发条件与执行体 |
| schema | Knowledge | 概念框架、分类法、数据模型 |
| heuristic | Knowledge | 启发式规则、经验法则 |
| case | Memory | 具体的案例或历史情景 |
| fact | Knowledge | 可验证的客观陈述 |
| state | Knowledge | 当前世界状态的快照 |

每个节点有 `strength`（强度，[0,1]）和 `health`（健康度，[0,1]）。

### 边关系模型

节点之间通过显式边连接，关系类型固定：

| 关系类型 | 含义 |
|---------|------|
| enables | 知识使能技能 |
| triggers | 状态/案例触发技能执行 |
| evidences | 案例为事实/规则提供证据 |
| contradicts | 矛盾关系 |
| strengthens | 成功执行强化关联 |
| weakens | 失败执行削弱关联 |

边也带有强度，随时间动态更新。

### 认知四季节律

支持按子树粒度独立执行：

- **春 · 生**：探索与播种，放宽过滤，鼓励试错
- **夏 · 长**：强化与茂盛，成功快速提升强度
- **秋 · 收**：反思与修剪，合并冗余，提炼高层知识
- **冬 · 藏**：休眠与重组，时间衰减，清除低强度节点

### 优雅关闭路径

1. 停止接收新请求 → 2. 等待运行中任务收敛 → 3. 刷新日志/告警/指标 → 4. 释放连接池

任何长时间运行的协程都要响应取消，检索和巡检在步骤之间检查 `cancel_requested`。

### DI 容器

`ServerContainer` 管理所有单例：config、log_service、async_mysql_pool、alert_service、http_client、yggdrasil_engine。

新增单例时考虑：生命周期（启动/关闭钩子）、依赖顺序、是否需要懒加载、关闭顺序是否会引发悬挂连接。

## 设计哲学

把项目分散的设计取向显式化，作为未来决策的依据：

- **确定性边界 + 认知灵活性**：用工程手段（事务、巡检、快照）框住自主进化的不确定性，而不是放任。认知自主性的提升应该伴随治理能力的提升
- **状态外置**：所有节点、边、强度、健康度都走 MySQL，进程内只持有运行时缓存。便于重启恢复、回滚、多实例部署
- **可观测优先**：request_id 必须贯穿，日志缓冲异步写，关键路径加指标。不为微小性能收益牺牲可观测性
- **YAGNI + 扩展点**：不为假想需求写代码，但在 Node / Edge / Retrieval / Season 这些主轴上保留清晰的扩展点。判断标准——这个扩展点能用"换一个实现"而不是"加一个分支"完成新需求吗
- **失败显式**：错误不静默吞掉，有害节点明确标记，污染传播可追踪，回滚有日志可审计
- **抽象与具体的对应**：核心抽象（Node / Edge / Retrieval / Season）的命名要用领域语言，避免泄漏实现细节
